RestFul-API設計記事

本記事について

この記事では、バックエンドプログラマーが必要とされる RESTful API に関する知識について簡単に説明します。

まず、RESTful API を正式に紹介する前に、まずは API が何か を理解する必要があります。

API とは？

API（Application Programming Interface） は、アプリケーションプログラミングインターフェースの略です。

バックエンド開発では、主な作業はフロントエンドや他のバックエンドサービスに対して API を提供することです。例えば、ユーザーデータの検索 API などです。

しかし、API はバックエンドシステムが公開するインターフェースだけでなく、フレームワークで提供されるメソッドも API の一部です。

理解を容易にするために、いくつかの例を挙げます：

  1. ある電子商取引サイトで特定の商品を検索すると、電子商取引サイトのフロントエンドはバックエンドで提供された商品検索関連の API を呼び出します。

  2. JDK を使用して Java プログラムを開発する場合、ユーザーの入力を読み取りたい場合は、JDK が提供する IO 関連の API を使用する必要があります。

API を単なるプログラム間の通信の橋と考えることができ、その本質は関数に過ぎません。また、API の使用方法もルールに従う必要があります（例：データ入力と出力の形式などは API のプロバイダーによって定義されます）。

RESTful API とは？

RESTful API は、REST （Representational State Transfer）に基づいて構築された API です。REST は具体的に何かについては後で説明します。

一般的に、RESTful API に関連する記事は非常に難解であることがよくあります。これは、REST に関連するいくつかの概念が理解しにくいためです。しかし、実際には、私たちが日常的に使用する RESTful API の知識は非常に簡単で要約しやすいです！

例えば、次の2つの API を見たら、それが何に使われるかがすぐにわかるはずです！これが RESTful API の強みです！

RESTful API は、URL+HTTP メソッドを見れば URL の目的がわかり、HTTP ステータスコードを見ればリクエストの結果がわかります。

私たちが開発プロセスで API を設計する際にも、少なくとも RESTful API の最も基本的な要件を満たす必要があります（例：インターフェースでは名詞を使用し、POST リクエストでリソースを作成し、DELETE リクエストでリソースを削除するなど、例：GET /notes/id：特定の id のメモの情報を取得します）。

REST の解説

REST は REpresentational State Transfer の略です。このフレーズを理解するのはやや難しいかもしれませんが、実際には REST の完全な名前は Resource Representational State Transfer であり、それを直訳すると “リソースがネットワークを介してある形式で状態を転送する” となります。もし理解が追いつかない場合は、以下を続けて読んでください。間違いなく次の説明で REST が理解できるはずです。

上記で触れた概念を個別に解説し、理解を深めていきます。実際、下記の概念を理解する必要はありませんが、次のセクションで説明する内容を理解するためには役立ちます。しかし、他の人と “RESTful API” について議論するためには、しっかりと理解することをお勧めします！

  • リソース（Resource）：私たちは実際のオブジェクトデータをリソースと呼びます。リソースは集合または個々のエンティティのいずれかです。例えば、私たちのクラスは集合形式のリソースを表し、特定のクラスは個々のリソースを表します。各リソースには特定の URI（Uniform Resource Identifier）が対応しており、そのリソースを取得する必要がある場合は、その URI にアクセスするだけです。例えば、特定のクラスを取得するには /class/12 にアクセスします。また、リソースにはサブリソースが含まれる場合もあります。例えば、/classes/classId/teachers：指定されたクラスのすべての教師の情報をリストアップします。

  • 表現形式（Representational）："リソース"は情報エンティティであり、それは多くの外部表現形式を持つことができます。"リソース"の具体的な表現形式を、json、xml、image、txt などといいます。

  • 状態転送（State Transfer）：最初にこの言葉を見たとき、多くの人は戸惑うかもしれません。内心で「これは一体何なのか？」と思うでしょう。REST における状態転送は、より多くがサーバー側のリソースの状態を記述します。例えば、HTTP 動詞を使用して（HTTP 動詞を使用して）リソースの状態を変更します。注：インターネット通信プロトコル HTTP プロトコルは、状態を保持しないプロトコルであり、すべてのリソースの状態がサーバー側に保存されます。

これらの説明を総合すると、RESTful アーキテクチャの特徴は次のようになります：

  1. 各 URI は一種のリソースを表します。

  2. クライアントとサーバー間で、このようなリソースの特定の表現形式が渡されます（例：json、xml、image、txt など）。

  3. クライアントは特定の HTTP 動詞を使用して、サーバー側のリソースを操作します。これにより “表現層の状態転送” が実現されます。

RESTful API の仕様

アクション

  • GET：サーバーから特定のリソースを取得するためのリクエスト。例：GET /classes（すべてのクラスを取得）

  • POST：サーバー上に新しいリソースを作成します。例：POST /classes（クラスを作成）

  • PUT：サーバー上のリソースを更新します（クライアントが更新後のリソース全体を提供します）。例：PUT /classes/12（ID が 12 のクラスを更新）

  • DELETE：サーバーから特定のリソースを削除します。例：DELETE /classes/12（ID が 12 のクラスを削除）

  • PATCH：サーバー上のリソースを更新します（クライアントが変更のあるプロパティを提供します）。あまり使われませんが、ここでは例を挙げません。

パス（インターフェイス名）

パスは API の具体的なウェブアドレスを表します。実際の開発では、以下の規則が一般的です：

  1. ウェブアドレスには動詞を含めることはできず、名詞のみを含める必要があります。また、API の名詞は複数形を使用する必要があります。 REST では、リソースはしばしばデータベース内のテーブルに対応し、データベース内のテーブルはすべて同じタイプのレコードの “コレクション” です。API の呼び出しがリソースを関連付けない場合（計算、翻訳などの操作）、動詞を使用することができます。例：GET /calculate?param1=11&param2=33。

  2. 大文字を使用しないでください。ハイフン - を使用し、アンダースコア _ を使用しないでください。 例えば、招待コードは invitation-code ではなく、invitation_code ではなく、invitation-code と書きます。

  3. API のバージョンをうまく活用してください。 API に重大な変更が加えられ、以前のバージョンと互換性がない場合は、URL を使用してバージョン管理を行うことができます。例：http://api.example.com/v1、http://apiv1.example.com。バージョンは数字である必要はありませんが、数字が最も一般的です。日付、季節などをバージョン識別子として使用することもできます。プロジェクトチームで合意したものを使用します。

  4. インターフェイス名は名詞を使用し、動詞を使用しないでください。 RESTful API 操作（HTTP メソッド）はリソース（名詞）を操作し、アクション（動詞）を実行しません。

RESTful の究極の HATEOAS

RESTful の最高傑作は HATEOAS ですが、実際のプロジェクトではほとんど使用されません。

上記は RESTful API の基本ですが、開発プロセスで最も容易に実践できるものでもあります。

HATEOAS（Hypermedia as the Engine of Application State）は、RESTful アーキテクチャの完全な表現です。それはリンクを使用して、クライアントがリソースとやり取りすることを可能にする一連の原則です。しかし、実際のプロジェクトではほとんど使用されず、理解も難しいため、ここでは簡単に説明しますが、実際の開発プロセスではあまり使われません。

実際には、RESTful API は Hypermedia を実現することが望ましいです。つまり、レスポンスにリンクを提供し、他の API メソッドに連携できるようにすることで、ユーザーがドキュメントを参照せずに次の手順を理解できるようにします。例えば、ユーザーが api.example.com のルートディレクトリにリクエストを送信した場合、次のようなレスポンスが返されます。

まとめ

上記は RESTful API の基本ですが、開発プロセスで最も重要な理解は、RESTful API の概念や標準を完全に理解することよりも、プロジェクトのニーズに合わせてリソースの設計とサービスの提供をより合理的に行うことです。

  • RESTful API は、HTTP プロトコルを使用してリソースにアクセスするための一般的な方法です。

  • バックエンドプログラマーは、RESTful API を使用して、フロントエンドや他のシステムと通信し、データを提供します。

  • RESTful API は、HTTP メソッド（GET、POST、PUT、DELETE など）と URL パスを使用してリソースを操作します。

  • API パスの設計は、名詞を使用し、リソースの階層構造を反映します。パスは大文字を使用せず、アンダースコアを使用しないでください。

  • クライアントとサーバー間の通信には、適切な HTTP ステータスコードを使用します。

  • HATEOAS は RESTful API の理論的最高傑作ですが、実際のプロジェクトではほとんど使用されません。

最後に、RESTful API の設計は非常に重要ですが、それを完全に理解するには時間がかかります。重要なのは、プロジェクトのニーズに合わせて柔軟に適用することです。
